How to Work
실제 입력 방식 — 이렇게 지시했습니다
① 화면 구조 지정
"효율/수익 지표 화면을 6 Row, 12 카드 구조로 만들어줘. Row 1: 좌) Gross Sales Efficiency (바 차트, %) 우) Net Sales Efficiency (바 차트, %) Row 2: 좌) CAC/LTV/LTV:CAC Ratio (콤비네이션 차트) 우) CAC Payback Period (라인, 단위: 개월)"
→ 레이아웃·차트 유형까지 텍스트로 명시
② 산출 로직 명시
"산출 로직도 함께 반영해줘. ARPA = sum(vf219) ÷ 고객 수 (vf219 ≠ 0인 고객 수 기준) LTV = ARPA ÷ 이탈률 이탈률 = (1 - N-1년매출 ÷ N-2년매출) × 100 ⚠ 이탈률 = 0이면 LTV / LTV:CAC / CAC Payback 모두 빈칸으로 처리"
→ DB 컬럼명 · 예외 처리까지 함께 전달
③ 디자인 스펙 전달 — AI Pro로 초안 작성
"색상 스펙은 아래와 같아. 라이트블루 #90CAF9 다크블루 #1E88E5 오렌지 #FF9800 퍼플 #7E57C2 그레이 #9E9E9E 사이드바: 딥 네이비 #1E2A3A~#2C3E50 현재 페이지 하이라이트: #17B9A6"
→ AI Pro로 디자인 시스템 초안 작성 → HEX 코드 수준까지 Claude에 전달하여 일관성 확보

Verification Cases
핵심 검증 사례 — 설계 단계에서 해결한 3가지
CASE 01
BO 기준 vs 솔루션 기준 수치 충돌
발견된 문제
같은 매출 지표가 화면마다 다른 수치로 표시 — 집계 기준이 혼재
↓
AI와 검토 내용
orderlist 기준 단일화 / BO 매핑 예외 케이스 목록화
✓ 개발 전 집계 기준 확정 → l_sap.orderlist 단일 소스 적용
CASE 02
LTV 산출 불가 케이스 — 이탈률 = 0
발견된 문제
이탈률이 0일 때 LTV = ARPA ÷ 0 → 무한대 또는 오류 발생
↓
AI와 검토 내용
예외 조건 설계 — 이탈률 0이면 LTV / LTV:CAC / Payback 모두 빈칸
✓ 시안 단계 예외 처리 확정 → 개발 스펙에 즉시 반영
CASE 03
EMS 상품 지표 산출 불가 처리
발견된 문제
RC/NRC 데이터 분리 불가 → Recurring 비중 약 9%로 LTV 신뢰성 없음
↓
AI와 검토 내용
11개 상품 산출 가능 / EMS · EMS(All) 2개 N/A 처리로 화면 설계 반영
✓ 상품별 산출 가능 여부 시안에 명시 → 개발 오해 사전 방지
