    § 3프로젝트 아키텍처 — 재설계본 v2 Fence-First

    베스트프랙티스 검증 결과를 반영한 재설계본. 기존 v1(데이터는 fence 안 · 버스/런타임은 fence 밖)의 보안 결함을 해소한다. 핵심 전환: fence를 넘는 것은 "데이터"가 아니라 "계약(Contract)"이다.

    3.1 재설계 원칙 (Fence-First)

    ① Fence-First기밀 DB·최종 런타임·반출 게이트를 모두 fence(사내 보안망) 안에. VDI를 쓰는 이유(기밀 DB + 보안 승인 AIPro)를 버스·런타임에도 일관 적용
    ② 사내 Private 버스Public이 아닌 사내 Private 레포를 버스로. 비즈니스 로직·스키마·샘플이 공개망으로 새지 않음
    ③ 계약만 반출Claude는 물리 데이터가 아니라 논리 데이터 계약만 있으면 UI를 만든다 → fence를 넘는 건 계약·합성샘플뿐
    ④ Build/Run 분리UI 제작(fence 밖)과 실데이터 연결·구동(fence 안)을 분리. 검증 단계에 fence 내 실구동 1회 필수
    ⑤ Pre-push 게이트반출 검사를 push 이후(GitHub)가 아니라 push 이전(VDI 내부)에 — 나가기 전에 차단

    3.5.1 L3 메시지 규약 — 시퀀스 다이어그램

    handoff 메시지 한 건의 전체 생애주기를 Claude · GitHub(handoff/) · AIPro 3주체 간 시간축(위→아래) 메시지 흐름으로 표현. 우측 색 배지는 상태머신(open→in_progress→done→confirmed) 전이를 나타낸다.

    폴링 방식 — 두 AI는 실시간 통신이 불가하므로, 각 환경에서 세션 시작 시 git pull → handoff/의 미완료 상태(open·in_progress) 파일을 스캔해 자신에게 온 요청을 처리한다. status 필드가 곧 큐의 상태이며, confirmed로 닫힌 메시지는 이력으로 남는다. 개선 과제 Actions로 신규 request 커밋 시 상대 환경에 알림(웹메일 등)을 보내는 이벤트 트리거 방식은 별도 검토.

    두 AI는 서로 다른 보안망에 있어 직접 통신이 불가능하다. GitHub는 ①양 환경이 모두 방화벽 승인된 유일한 공유 채널이고 ②커밋·PR로 모든 변경이 이력으로 남아 3인·양 AI가 동일 상태를 참조할 수 있다. 그래서 저장소인 동시에 Claude↔AIPro 소통 버스로 쓴다.
