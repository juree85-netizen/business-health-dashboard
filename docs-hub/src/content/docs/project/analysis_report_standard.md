## 1. 목적 및 적용 범위

이 기준은 Claude가 내부 분석보고서를 작성할 때 Microsoft Word 기반 서식과
분석보고서 품질 기준을 함께 준수하도록 지시하기 위한 실행 템플릿이다.

**적용 대상:**
- 내부 업무 분석보고서
- 경영진 또는 실무 조직 대상 상황 분석 보고서
- 이슈, 원인, 영향, 대응 방향을 정리하는 문서
- 의사결정 지원을 위한 근거 기반 보고서

이 기준은 시각 서식만 정의하지 않는다. 좋은 분석보고서가 갖춰야 할
논리, 근거, 사실/해석/제안 구분, 검토 기준까지 함께 정의한다.

## 2. 분석보고서 품질 원칙

### 2.1. 결론 중심
- 핵심 결론은 제목, 요약, 본문 초반에서 빠르게 파악되어야 함
- 배경 설명은 결론과 판단에 필요한 범위로 제한
- 독자가 어떤 판단이나 행동을 해야 하는지 분명히 드러냄

### 2.2. 사실·해석·제안의 분리
- 확인된 사실은 단정형으로 작성
- 해석은 "판단됨", "가능성이 있음", "시사함"처럼 추론임을 표기
- 권고안이나 후속 조치는 사실 및 해석과 구분하여 작성
- 근거가 부족한 내용은 "가정" 또는 "확인 필요"로 표시

### 2.3. 근거 기반 작성
- 숫자, 날짜, 조직명, 금액, 고유명사는 원자료 기준으로만 사용
- 원자료에 없는 정보는 임의로 보완하지 않음
- 필요한 정보가 없으면 보고서 안에 "확인 필요"로 분리 표기
- 중요한 주장에는 근거, 출처, 관찰 사실, 비교 기준 중 하나 이상 연결

### 2.4. 실행 가능성
- 분석 결과는 가능한 한 실행 가능한 후속 조치와 연결
- 권고안에는 책임 주체, 우선순위, 예상 효과, 유의사항 포함
- 불확실성이 큰 경우 즉시 실행안보다 추가 확인 항목 제시

## 3. 문체 및 표기 기준

### 3.1. 어조
- 매우 건조하고 공식적인 행정/정책 보고서 스타일 유지
- 감정적 표현, 홍보성 표현, 구어체, 과장 표현 배제
- 판단 내용은 간결하게 쓰되, 근거와 원인-결과 관계가 드러나게 작성

### 3.2. 명사 우선·명사형 종결

- 모든 문장과 항목의 끝은 **명사 또는 명사형 종결**로 통일
- 명사를 우선하고, 의미가 어색한 경우에만 명사형 종결 허용
- **서술형 종결(`~한다`, `~하였다`, `~입니다`)은 원칙적으로 사용 금지**
- **문장 끝의 마침표는 사용하지 않음**

| 구분 | 예시 |
|------|------|
| 명사 우선 (권장) | `추가 확인 필요` / `검토 대상 아님` / `자료 미제출` |
| 명사형 허용 | `~함` / `~됨` / `~판단됨` |
| 금지 | ~~`추가 확인이 필요하다.`~~ / ~~`검토 대상이 아닙니다.`~~ |

### 3.3. 문장 구조
- 시계열 흐름과 원인-결과 관계를 명확히 하는 구조 선호
- 필요한 경우 `'13年 → '18年`처럼 기간 흐름을 직접 표시
- "배경 → 변화 → 영향 → 판단"의 밀도 있는 복문 구조 우선

### 3.4. 한자 직접 표기

중요한 명사, 조직, 역할, 방향, 범위를 나타낼 때 아래 한자를 직접 사용한다.
한글 뒤 괄호 병기 방식이 아니라, 필요한 위치에 한자를 직접 쓴다.

| 한글 | 한자 | 사용 예 |
|------|------|---------|
| 미국 | 美 | 美 연준, 美 클라우드법 |
| 내 | 內 | 사업부 內 |
| 외 | 外 | 대외 협력 外 |
| 등 | 等 | Blackstone 等 |
| 월 | 月 | '26.5月, 月간 보고 |
| 연 | 年 | '21年, 3年 내 |
| 때·시 | 時 | 인수 時, 적용 時 |
| 없음 | 無 | 실적 無 |
| 있음 | 有 | 경고 신호 有 |
| 중 | 中 | 협상 中 |
| 전 | 前 | 상장폐지 前 |
| 전체 | 全 | 全 부서 |

### 3.5. 영문 병기
- 법률 명칭, 기업명, 전문 개념, 원문 용어는 필요 시 괄호 안에 영문 병기
- 이미 널리 통용되는 약어는 최초 1회만 병기
- 예: `美 클라우드법(CLOUD Act)` / `관할권(jurisdiction)` / `LBO(Leveraged Buy-Out)`

### 3.6. 날짜 표기
- 연도+월: **`'YY.M月`** 형식 → `'26.5月`, `'18.3月`
- 보고 날짜: **`'YY.MM.DD`** 형식 → `'26.05.08`
- 연도만: 필요 시 **`'YY年`** 형식 → `'21年`, `'18年`

## 5. Microsoft Word 문서 디자인 스펙

### 5.1. 문서 기본 설정

| 항목 | 기준값 |
|------|--------|
| 용지 | A4 세로 (210 × 297mm) |
| 여백 | 좌/우/상/하 **20mm** |
| 기본 폰트 | **바탕체(Batang) 14pt** |
| 줄간격 | **1줄 (Single)** |
| 제목 폰트 | 바탕체 **20pt**, 굵게, 밑줄, 가운데 정렬 |
| 페이지 번호 폰트 | 바탕체 **12pt** |

### 5.2. 머리말 구성

```
[제목]         바탕체 20pt, 굵게, 밑줄, 가운데 정렬  (단락 뒤 12pt)
[보고 날짜]    바탕체 14pt, 보통, 오른쪽 정렬. 예: '26.05.08  (단락 뒤 9pt)
[요약 표]      1행 3열 표 (5.3 참조)
```

### 5.3. 요약 표 (1행 3열)

```
┌───┬─────────────────────────────────────────────────┬───┐
│   │  (Space 1칸) 요약 1줄 — 분석 대상과 목적 중심    │   │
│   │  (Space 1칸) 요약 2줄 — 핵심 원인 또는 발견사항  │   │
│   │  (Space 1칸) 요약 3줄 — 필요한 조치 또는 권고안  │   │
└───┴─────────────────────────────────────────────────┴───┘
 ↑                         ↑                           ↑
좌측 셀                  가운데 셀                    우측 셀
폭 ~7pt                 폭 ~165mm                   폭 ~7pt
검은선(상/좌/하)         테두리 없음                  검은선(상/우/하)
```

**세부 규칙:**
- 표 자동 맞춤 해제 (fixed layout)
- 좌우 셀: 폭 7pt(140 twips), 셀 여백 0
- 가운데 셀: 폭 165mm(9353 twips), 셀 좌우 여백 1.4pt(28 twips), 상하 0pt
- 좌측 셀 테두리: 상/좌/하 검은 단선, 우측 없음
- 가운데 셀 테두리: 4면 모두 없음
- 우측 셀 테두리: 상/우/하 검은 단선, 좌측 없음
- 가운데 셀: 3개 문단, 각 선행 Space 1칸, 단락 뒤 9pt
- 넘버링, `요약:` 제목, Lv1 기호 없음

### 5.4. 본문 계층 구조

**모든 들여쓰기는 Space 문자로만 처리한다. Word의 `left_indent` / `hanging_indent` / `first_line_indent` / 탭 문자 사용 금지.**

| 레벨 | 기호 | 스타일 | 기호 시작 | 줄간격 | 단락 앞 | 단락 뒤 |
|------|------|--------|----------|--------|--------|--------|
| Lv1 | `1.` | 14pt, **굵게** | 0칸 | 1줄 | **16pt** | **12pt** |
| Lv2 | `□` | 14pt, **굵게** | 1칸 | 1줄 | 0pt | 9pt |
| Lv3 | `-` | 14pt, 보통 | 3칸 | 1줄 | 0pt | 9pt |
| Lv4 | `·` | 14pt, 보통 | 5칸 | 1줄 | 0pt | 9pt |
| Lv5 | `※` | 14pt, 보통 | 3칸 | 1줄 | 0pt | 9pt |

**Lv5는 계층 심화가 아니라 보충 주석 레벨 → Lv3과 동일한 기호 시작 위치 사용**

텍스트 표현 예시:
```
1. 소제목 내용

 □ 주요 항목 내용

   - 세부 항목 내용

     · 상세 내용

   ※ 보충 주석 내용
```

### 5.5. 계층 작성 규칙

- 하위 레벨을 사용할 때는 상위 레벨을 먼저 제시
- 레벨을 건너뛰지 않음 (Lv1 바로 아래 Lv3 금지)
- 기호(`□`, `-`, `·`, `※`, `1.`)를 텍스트에 직접 입력
- Word 자동 번호 매기기·자동 글머리 기능에 의존하지 않음
- 하이픈(`-`)이 대시로 자동 변환되지 않도록 일반 하이픈 사용

**Enter 분리(이어쓰기) 규칙:**
- 글머리 항목이 길어져 별도 문단으로 분리할 때 기호를 반복하지 않음
- 이어쓰기 문단 선행 Space: Lv2=5칸, Lv3=6칸, Lv4=7칸

**번호형 나열:**
- 동일 수준 나열 시 `1) 2) 3)` 또는 `① ② ③` 사용 가능
- `- 1)`, `· 1)`처럼 글머리 기호와 번호형 기호를 함께 쓰지 않음
- 번호형 나열은 해당 레벨의 기호를 대체함

**강조 표기:**
- 결론·영향·시사점·핵심 판단은 `→` 기호로 표시 가능
- `→` 강조 문장: 바탕체 14pt, 굵게 또는 굵게+밑줄 허용
- `→` 강조는 핵심 판단에만 한정하고 남용 금지

### 5.6. 마무리

- 페이지 번호: 페이지 하단 중앙에 **`- N -`** 형식 (바탕체 12pt)
- 문서 끝 표시: 본문 마지막 줄에 오른쪽 정렬로 **`- 以 上 -`** 삽입 (바탕체 14pt)

## 6. Claude 생성 지침

### 6.1. 생성 전 확인사항

보고서를 생성하기 전에 아래 입력 정보를 확보한다.

```
- 보고서 제목:
- 보고 대상:
- 작성 목적:
- 핵심 질문:
- 원자료:
- 분석 기간/범위:
- 반드시 포함할 내용:
- 제외할 내용:
- 원하는 결론 또는 관점:
- 분량:
- 마감 또는 검토 일정:
- 미확인 사항:
```

필수 정보가 부족하면 먼저 질문하거나, 본문에 "확인 필요"로 분리 표기한다.

### 6.2. Claude 생성 순서

1. 제목 단락 생성 (20pt, 굵게, 밑줄, 가운데)
2. 보고 날짜 단락 생성 (`'YY.MM.DD 소속`, 14pt, 오른쪽)
3. 요약 표 생성 (1행 3열, 5.3 규칙 준수)
4. Lv1~Lv5 계층 본문 생성 (5.4~5.5 규칙 준수)
5. 표가 필요한 곳에 데이터 표 삽입
6. 끝 표시 (`- 以 上 -`, 오른쪽)

### 6.3. Claude 흔한 오류 목록

아래 오류는 Claude가 자주 범하는 항목이다. 생성 후 반드시 자가 검증한다.

| 오류 유형 | 금지 | 올바른 표현 |
|-----------|------|------------|
| 서술형 종결 | `~한다`, `~하였다`, `~입니다` | `~함`, `~필요`, `~확인` |
| 마침표 | 항목 끝 `.` | 마침표 없음 |
| 폰트 혼용 | 맑은 고딕, 나눔고딕 사용 | 바탕체(Batang) 14pt |
| Word indent 속성 | `left_indent=Cm(2)` 등 | 선행 Space 문자로 대체 |
| 날짜 형식 | `2026-05-08`, `26.5.8` | `'26.05.08` |
| 요약 박스 | 4면 테두리 박스 단락 | 1행 3열 표 구조 |
| Lv 건너뜀 | Lv1 바로 아래 Lv3 배치 | Lv1 → Lv2 → Lv3 순서 |
| 글머리+번호 중복 | `- 1)`, `· ①` | `1)` 또는 `-` 하나만 |
| 원자료 부재 수치 | 임의 수치 보완 | "확인 필요"로 표기 |
| 끝 표시 형식 | `— 以 上 —` | `- 以 上 -` |

### 6.4. 행말 자간 보정 (사람만 수행)

행말 자간 보정은 Word에서 사람이 직접 수행하는 작업으로 Claude가 생성 단계에서 처리할 수 없다.
Claude는 내용 누락 없이 의미 단위별로 분리하는 방식으로 Enter 분리 문단을 생성하고,
시각 보정은 사람이 Word에서 확인한다.

## 9. 검증 기준

### 9.1. Claude 자가 검증 (생성 직후)

생성 후 아래 항목을 순서대로 확인한다. 미통과 항목이 있으면 즉시 수정한다.

**서식 검증:**
- [ ] 바탕체 14pt 적용 여부 (제목 20pt, 페이지 번호 12pt 예외)
- [ ] Lv1 단락 앞 16pt / 뒤 12pt, Lv2~Lv5 뒤 9pt 적용 여부
- [ ] 들여쓰기가 Space 문자로만 처리되었는지 (Word indent 속성 없음)
- [ ] 요약 표가 1행 3열 구조로 생성되었는지
- [ ] 줄간격이 SINGLE로 설정되었는지

**문체 검증:**
- [ ] 모든 항목 끝이 명사 또는 명사형 종결인지
- [ ] 항목 끝에 마침표가 없는지
- [ ] `~한다`, `~합니다`, `~입니다` 서술형이 없는지
- [ ] 날짜가 `'YY.MM.DD` 또는 `'YY.M月` 형식인지

**내용 검증:**
- [ ] 핵심 결론이 요약에 포함되는지
- [ ] 사실·해석·제안이 혼재하지 않는지
- [ ] 원자료에 없는 수치나 고유명사를 임의 보완하지 않았는지
- [ ] 미확인 사항이 "확인 필요"로 표시되었는지

**마무리 검증:**
- [ ] 페이지 번호 형식이 `- N -`인지
- [ ] 문서 끝이 `- 以 上 -`이고 오른쪽 정렬인지
- [ ] 특수문자(`□`, `·`, `※`, `→`, `以 上`)가 깨지지 않았는지

### 9.2. 시각 검증 (사람 수행)

아래 항목은 Word 또는 렌더링 환경에서 사람이 직접 확인한다.

- [ ] 요약 표 좌우 검은선 셀이 세로선처럼 보이는지
- [ ] 가운데 셀 테두리가 보이지 않는지
- [ ] 들여쓰기 공백이 레벨별로 시각적으로 일치하는지
- [ ] 줄간격이 1줄 기준으로 균일하게 보이는지
- [ ] 폰트 크기가 14pt로 일관되어 보이는지
- [ ] 특수문자와 한자가 깨지지 않고 표시되는지

### 9.3. 완료 조건

- Claude 자가 검증 전 항목 통과
- 미확인·미통과 항목 없음
- 시각 검증 대상 항목은 "시각 검증 필요"로 별도 표기하거나, 사람 확인 후 통과 기록

## 10. 인코딩 및 문자 보존

- 문서는 UTF-8로 저장
- 아래 특수문자가 깨지지 않아야 함:
  `□`, `·`, `※`, `①②③`, `→`, `- 以 上 -`,
  `美`, `社`, `內`, `外`, `等`, `月`, `年`, `時`, `無`, `有`, `可`, `前`, `全`, `中`
- Word에 붙여넣은 뒤 특수문자와 Space 들여쓰기가 유지되었는지 최종 확인
