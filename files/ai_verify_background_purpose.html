01추진 배경
CURRENT STATE 사업건전성 대시보드는 현재 Tableau + Impala 기반으로 운영 중이며,
요구사항 변경 시마다 웹디자이너·Tableau 개발자·DB 개발자 등 전문 인력이 필요하다.
🎨
웹디자이너 리소스
화면 변경·신규 시안마다 별도 디자인 리소스 확보 필요
📊
Tableau 개발자 리소스
지표 추가·대시보드 구조 변경 시 Tableau 개발 인력 의존
🗄️
DB 개발자 리소스
데이터 모델·쿼리 변경마다 DB 전문 인력 필요
문제 — 전문 인력 리소스 과다 소요로 요구사항이 적기에 반영되지 못하는 구조적 지연이 반복됨.
02추진 목적
PURPOSE 전문가(웹디자이너·Tableau 개발자·DB 개발자) 없이, 기획자가 AI 도구만으로
대시보드를 구축 → 운영 → 유지보수할 수 있는지 검증한다.
AS-IS
  • Tableau 기반 시각화
  • Impala 데이터 레이어
  • 전문가팀(디자이너·개발자·DBA) 의존
→
TO-BE
  • Web / WAS 자체 구현
  • 필요 시 추가 애플리케이션 확장
  • Impala는 데이터 레이어로 유지
※ Tableau보다 더 나은 대안이 있는지, 전환 시 보안 요건을 충족하는지는 검증 과정에서 별도 검토가 필요하다.

06환경설정 고려사항
  • 1워크스페이스 구조 — 레포 내 디렉터리 구조와 역할 분리 기준 확립plan / work / handoff / verify 등 단계별 산출물 분리
  • 2인덱스 — 레포 전체 현황을 한눈에 보는 색인 문서 상시 갱신
  • 3문서 포맷 — Human readable / Machine readable 동시 고려json / yaml / python 등 기계 판독 가능한 형식 활용
  • 4스냅샷·버전 관리 — 주요 산출물 시점별 스냅샷·버전 관리 체계
  • 5CLI 대화 지속성 — claude.md · agent.md · skill.md 기준으로 로그인 시 이전 대화 이어가기
  • 6컨텍스트 과다 방지 — 대화·문서 컨텍스트가 과도하게 누적되지 않도록 관리
  • 7Project Preview Hub — 산출물을 미리보기로 확인할 수 있는 허브 운영
  • 83인 충돌·누락 방지 — 동시 작업 시 충돌·누락을 방지하는 협업 규약
  • 9Public GitHub 활용 — 검증 단계 산출물의 Public 레포 활용 방안
  • 10검증 → 운영 전환 용이성 — 검증 환경 산출물이 실 운영 서버 전환 시 재사용 가능하도록 설계
  • 11업무 분장 3역할① 기획·환경·DevOps·보안  ② 사용자·화면·지표·Frontend  ③ Backend·데이터분석·모델링
  • 12Impala c_tableau 규칙 — 기존 테이블·타 스키마 변경·삭제 금지, 검증용 테이블만 사용AIPro는 서버·테이블·컬럼 정보를 변환하여 공유
